# Streamline — Requirements Document
**MVP v1** | Project management for non-project managers

---

## Overview

Streamline is a simplified project management tool designed for people who manage projects as part of their role, but aren't formal project managers. The focus is on clarity, simplicity, and removing the overhead that makes tools like Jira or Asana feel heavy for everyday use.

---

## 1. Authentication & Organization

| ID | Requirement | Description |
|----|-------------|-------------|
| AUTH-01 | Email & password sign-up | Users register with an email address and a chosen password. Standard credential-based auth via Supabase Auth. |
| AUTH-02 | Login & session management | Returning users log in with email/password. Supabase handles JWT sessions and token refresh. |
| ORG-01 | Organization creation | The first user to set up an org becomes its admin. The org has a name and a unique identifier. |
| ORG-02 | Join organization on onboarding | New users can join an existing org during sign-up via an invite link generated by an admin. |
| ORG-03 | Roles: admin & member | Admins manage org settings, users, teams, and all projects. Members can view and edit projects they're assigned to. |

---

## 2. Invite by Link

| ID | Requirement | Description |
|----|-------------|-------------|
| INV-01 | Admin generates invite link | An org admin can generate a unique invite link from within the app. The link encodes a random token stored in Supabase with the org ID and an expiry timestamp. |
| INV-02 | Token-based org join | When a new user registers via an invite link, the app reads the token from the URL, validates it (not expired, not already used), and automatically adds the user to the correct org. |
| INV-03 | Token expiry & single use | Invite tokens expire after a set period (e.g. 7 days) and are invalidated after first use to prevent link sharing beyond the intended recipient. |
| INV-04 | Admin-managed distribution | Admins copy and share the link themselves via Slack, email, or any channel. No email sending is required from the app. |

---

## 3. Teams

Teams are a first-class object in Streamline. Users belong to teams, and projects are assigned to teams. This ensures that filtering the Gantt by team or by person works consistently — both are membership-based queries.

| ID | Requirement | Description |
|----|-------------|-------------|
| TM-01 | Team creation | Admins can create named teams within an org (e.g. Engineering, Marketing, Operations). |
| TM-02 | Team membership | Users are assigned to one or more teams. Team membership is managed by admins. |
| TM-03 | Project-to-team assignment | Projects are assigned to a team at creation time and can be reassigned later. |
| TM-04 | Team management view | Admins can view and edit team composition from an org settings area. |

---

## 4. Projects

| ID | Requirement | Description |
|----|-------------|-------------|
| PRJ-01 | Create & manage projects | Users can create projects with a name, description, owner, team, start date, and target completion date. |
| PRJ-02 | Project detail view | Drill-in view showing all project metadata: start date, target & actual completion dates, budget, status, and assigned members. |
| PRJ-03 | Project status tracking | Projects carry a status (not started, in progress, complete, on hold) that can be updated manually. |
| PRJ-04 | Budget field | Projects can have an allocated budget value recorded for reference. |

---

## 5. Tasks & Action Items

The hierarchy is: **Project → Task → Action Item**. Action items are lightweight subtasks within a task — checklist-style items for granular work tracking. If action items later need their own assignees or due dates, they can be promoted to full tasks.

| ID | Requirement | Description |
|----|-------------|-------------|
| TSK-01 | Tasks within projects | Projects contain tasks with a name, assignee(s), start date, due date, and status. |
| TSK-02 | Action items (subtasks) | Tasks can contain action items — checklist-style subtasks for tracking specific pieces of work within a task. |
| TSK-03 | Task assignment | Tasks can be assigned to one or more org members. Assignees can see their tasks across projects. |
| TSK-04 | Task status tracking | Tasks have their own status (not started, in progress, complete, blocked) independent of the parent project status. |

---

## 6. Gantt Chart View

| ID | Requirement | Description |
|----|-------------|-------------|
| GNT-01 | Company-wide Gantt | Top-level view showing all active projects across the org on a timeline. |
| GNT-02 | Filter by team | Gantt can be filtered to show only projects belonging to a specific team. |
| GNT-03 | Filter by person | Gantt can be filtered to show only projects or tasks assigned to a specific individual. |
| GNT-04 | Task-level bars | Projects can be expanded on the Gantt to show individual tasks as sub-bars within the project row. |

> **Implementation note:** Use open source Frappe Gantt library rather than building a custom Gantt renderer. This saves significant development time and is the right call for a hackathon timeline.

---

## 7. Infrastructure

| ID | Requirement | Description |
|----|-------------|-------------|
| INF-01 | Supabase backend | All data stored in Supabase (Postgres). Row Level Security (RLS) enforces org-scoped data access so users can only see data within their own org. |
| INF-02 | Supabase Auth | Authentication handled by Supabase Auth. Role and profile info stored in a custom `profiles` table linked to `auth.users`. |

---

## Data hierarchy summary

```
Organization
├── Teams
│   └── Members (users)
└── Projects (assigned to a team)
    └── Tasks (assigned to members)
        └── Action Items (subtasks)
```

---

## Out of scope for MVP v1

The following features are explicitly deferred to avoid scope creep during the hackathon:

- Email notifications
- Comments on tasks or projects
- File attachments
- Reporting / analytics dashboards
- SSO / OAuth login (Google, GitHub, etc.)
- Mobile app
- Time tracking
- Custom fields on projects or tasks
